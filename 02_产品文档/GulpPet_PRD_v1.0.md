## 一、产品背景

### 1.1 问题定义

长期在电脑前工作的知识工作者（职场人、学生）普遍存在**忘记喝水**的问题，原因并非不知道喝水重要，而是：

- **注意力被工作占据**，口渴信号容易被忽略
- **现有提醒方式干扰性强**，通知一多就习惯性关掉
- **记录门槛高**，精确记录毫升数让人烦躁，久而久之放弃

市面上现有产品（WaterMinder、Plant Nanny 等）的核心问题是：**手机通知在用户专注工作时几乎无效**，且记录交互过于繁琐，导致用户留存率低。

### 1.2 解决方案思路

GulpPet 的核心差异点：

- **桌宠而非通知**：提醒以桌面悬浮宠物的状态变化呈现，不打断工作流，让用户主动关注而非被动打扰
- **时间驱动而非目标驱动**：提醒频次基于用户工作时段均匀分布，与"喝了多少"解耦，算法简洁可靠
- **一键记录**：预设杯型 + 上次使用优先，单次记录不超过 2 次点击
- **双端无缝同步**：离桌时手机接管，数据实时同步，提醒不断档

### 1.3 产品定位

> **懂你工作节奏的补水小伙伴**

面向长期使用电脑工作或学习的知识工作者，通过桌面灵宠的陪伴形式，以低打扰、高感知的方式帮助用户建立科学补水习惯的双端应用。

### 2.1 核心用户群

| 维度 | 描述 |

| 职业 | 知识工作者（互联网、设计、学术、学生等） |

| 痛点 | 专注工作时忘记喝水，口渴时才意识到已经很久没补水 |
| 动机 | 关注健康、希望养成好习惯，但不想被繁琐工具打扰 |

### 2.2 用户画像（Persona）

**Persona A — 主流用户**

> 李晓雯，26岁，互联网运营，上海
> 每天对着电脑 9 小时，桌上放着 500ml 保温杯，但下班前经常才喝了一半。
> 用过 WaterMinder，但手机通知太烦直接关掉了。
> 她需要的不是强提醒，而是一个在她工作时陪着她、不打扰她、但让她"想起来"的东西。

**Persona B — 边缘用户（自我管理型）**

> 陈浩，32岁，自由职业设计师，在家办公

> 对桌宠的形式感兴趣，但核心诉求是数据可视化——知道自己今天喝了多少。

## 三、产品目标与成功指标

### 3.1 产品目标

**用户目标：** 帮助用户在不改变工作习惯的前提下，养成每天按时补水的习惯。

**产品目标（MVP 阶段）：**
- 验证"桌宠提醒"形式是否比传统通知有更高的留存率
- 验证"时间驱动提醒 + 一键记录"是否足够轻量，让用户愿意长期使用

### 3.2 指标体系

#### 北极星指标

**连续 7 日目标完成率**
= 连续 7 天每日饮水量 ≥ 日目标 80% 的用户比例

> 选择理由：这一时刻代表用户真正通过产品建立了补水习惯，是产品核心价值的实现。

#### 核心指标

| 指标 | 定义 | 目标值（MVP 参考） |
|---|---|---|
| 7日留存率 | 第1天使用后第7天仍活跃的用户比例 | ≥ 40% |
| 提醒响应率 | 收到提醒后点击"已喝水"的比例 | ≥ 50% |
| 日均记录次数 | 用户每天主动记录饮水的平均次数 | ≥ 4 次 |
| 初始化完成率 | 打开 app 后完成全部初始化设置的比例 | ≥ 85% |

#### 护栏指标

| 指标 | 定义 | 警戒线 |
|---|---|---|
| 提醒关闭/屏蔽率 | 用户主动关闭提醒功能的比例 | ≤ 15% |
| 提醒"跳过"率 | 每次提醒中选择"跳过"的比例 | ≤ 30% |
| 设置页退出率 | 进入设置后未完成修改直接退出的比例 | ≤ 20% |

## 五、功能详细说明

### 5.1 初始化引导流程

**触发时机：** 首次安装 / 账号初次登录

**流程步骤：**

```
步骤1：设备选择
  ┌─────────────────────────────────────┐
  │  你主要用哪些设备？                   │
  │  ● 桌面 + 手机（推荐）🐾 含桌宠功能  │  ← 视觉重点
  │  ○ 仅桌面端          🐾 含桌宠功能  │
  │  ○ 仅手机端                          │  ← 弱化展示
  └─────────────────────────────────────┘

步骤2：基础信息
  - 性别（男 / 女 / 不透露）
  - 年龄段（18–49岁 / 50–64岁 / 65岁及以上）
  - 活动强度（三个场景描述卡片选一）

步骤3：作息设置（仅桌面端或双端用户）
  - 主要使用电脑的时段：开始时间 + 结束时间
  - 说明文案："我们只在这段时间内提醒你，不在线时段不打扰"

步骤4：杯型设置
  - 从预设杯型中选择你最常用的（可多选，最多5个）
  - 可自定义命名 + 输入毫升数
  - 预设选项：一次性杯200ml / 玻璃杯·马克杯350ml / 保温杯500ml / 矿泉水瓶550ml

步骤5：选择桌宠形象 ✨
  - 展示全部 6 款宠物供用户选择（宫格布局）
  - 鼠标悬停时预览该宠物的待机动画
  - 选中后右侧展示该宠物的名称 + 简短性格描述（1句话）
  - 点击"确认，让它出现！"→ 宠物立即出现在桌面上
  - 说明文案："随时可以在设置中更换它"

步骤6：完成
  - 展示计算出的每日目标（X ml）
  - 展示提醒间隔（大约每 X 分钟提醒一次）
  - 引导用户将应用设为"登录时自动启动"
```

**目标饮水量计算逻辑（对应科学依据文档）：**

| 输入 | 基础值（中国 DG 2022） | 活动强度调整 |
|---|---|---|
| 男性 | 1700 ml | 中度 +400 ml / 重度 +700 ml |
| 女性 | 1500 ml | 中度 +400 ml / 重度 +700 ml |
| 65岁以上 | max(基础值, 1500 ml) | 同上 |

### 5.2 桌面端 — 灵宠浮窗

**基本形态：**
- 常驻桌面的透明悬浮窗，宽约 120px，高约 160px
- 始终置顶（Always-on-Top），不遮挡鼠标点击穿透（Click-through）
- 可鼠标拖拽到屏幕任意位置，位置持久化存储

**宠物状态（S1 功能）：**

| 状态 | 触发条件 | 宠物表现 |
|---|---|---|
| 正常 | 今日进度正常，距下次提醒 > 10 分钟 | 待机动画（轻微晃动） |
| 缺水中 | 已到提醒时间，等待用户响应 | 渴望动画（更明显的动作） |
| 今日达标 | 饮水量 ≥ 日目标 | 开心动画 |

**动画实现方案：Lottie**

- 技术方案：`lottie-react` 库播放 `.json` 动画文件，Electron 窗口设置 `transparent: true` 实现透明背景
- 素材来源：从 [LottieFiles.com](https://lottiefiles.com) 选取免费可商用动画，每种状态对应一个 `.json` 文件
- 状态切换：根据宠物当前状态动态切换播放的动画文件，无需重载组件
- 文件结构：
  ```
  assets/animations/
    idle.json       ← 正常待机
    thirsty.json    ← 缺水提醒中
    happy.json      ← 今日达标
  ```

**右键菜单（单击宠物）：**
- 查看今日进度
- 记录饮水（快速入口）
- 更换形象 → 打开桌宠画廊
- 设置
- 退出

### 5.5 桌面端 — 智能提醒系统

#### 提醒间隔算法

```
基础间隔 = 工作时段总分钟数 ÷ 8
约束：max(30, min(90, 基础间隔))

示例：
  09:00–18:00 → 540分钟 ÷ 8 = 67.5分钟 → 68分钟
  20:00–24:00 → 240分钟 ÷ 8 = 30分钟 → 下限30分钟

65岁以上特殊模式：间隔缩短15%
```

#### 提醒弹窗设计

```
┌──────────────────────────────┐
│  🐾 咕嘟提醒你喝水啦～         │
│  距上次喝水已过去 68 分钟       │
│                              │
│  建议补充：一杯水（约350ml）    │
│                              │
│  [已喝水 ✓]  [稍后15分钟]  [跳过] │
└──────────────────────────────┘
```

- 弹窗位置：屏幕右下角，不遮挡主工作区
- 弹窗自动消失时间：30 秒后若无操作，等同于"跳过"
- 全屏模式：自动检测全屏状态，进入全屏时提醒静默，退出全屏后恢复

#### 用户响应后的计时重置逻辑

| 操作 | 下次提醒时间 | 饮水记录 |
|---|---|---|
| 已喝水（选择杯型） | 记录时刻 + 基础间隔 | 记录该杯型容量 |
| 稍后15分钟 | 当前时刻 + 15分钟 | 不记录 |
| 跳过 | 当前时刻 + 基础间隔 | 不记录 |
| 30秒无操作自动消失 | 当前时刻 + 基础间隔 | 不记录 |

#### 免打扰规则

| 触发条件 | 行为 |
|---|---|
| 检测到应用进入全屏（如 Keynote 演示） | 提醒静默，计时器暂停 |
| 用户在设置中配置的免打扰时段 | 跳过该时段所有提醒 |
| 用户手动开启"专注模式" | 暂停所有提醒，系统托盘显示🔕 |
| 桌面应用关闭 | 计时暂停，手机端接管 |

### 5.10 手机端 — 功能范围（V1 轻量版）

V1 手机端定位为**桌面端的补充**，不做成独立完整应用，聚焦三个核心场景：

- 用户离开电脑后继续收到提醒
- 在手机上快速记录一次饮水
- 查看今日进度

**手机端页面结构：**

```
底部导航（3个Tab）：
  🏠 首页（今日进度 + 快速记录）
  📊 记录（时间线）
  ⚙️ 设置（同步自桌面端设置）
```

**首页（Today）：**
- 环形进度图：已喝 / 目标
- 下次提醒倒计时（离桌后由手机接管）
- 快速记录按钮（弹出杯型选择底部弹窗）
- 咕嘟宠物状态展示（简化版，静态图）

### 5.11 数据同步逻辑

**技术方案：** Firebase Firestore（实时数据库）

**数据架构：**

```
users/{userId}/
  ├── profile（用户设置，初始化信息）
  ├── dailyGoal（今日目标 ml）
  ├── cups（杯型列表）
  └── events/{date}/{eventId}（HydrationEvent）

HydrationEvent {
  kind: "suggestion" | "response" | "measurement"
  amountMl?: number
  source: "desktop" | "mobile" | "manual"
  timestamp: string (ISO 8601)
}
```

**同步规则：**
- 所有 HydrationEvent 写入 Firestore，双端实时监听
- 桌面端和手机端均可写入 event，Firestore 负责合并
- 离线时本地缓存，联网后自动同步（Firestore 原生支持）
- 今日进度由客户端实时计算（sum of amountMl where kind = "response" or "measurement"，today's events）

## 六、核心用户流程

### 6.1 首次使用流程

```
安装应用
  → 打开 → 初始化引导（5步）
  → 系统计算每日目标 + 提醒间隔
  → 进入主界面（桌宠出现在桌面）
  → 第一次提醒（约 间隔时间 后）
  → 用户响应 → 记录饮水
  → 计时器重置
```

### 6.2 日常使用流程（桌面端）

```
用户工作中
  → [间隔时间到] 提醒弹窗出现（右下角）
  → 三个操作分支：
      A. 已喝水 → 选杯型 → 记录 → 弹窗关闭 → 计时器重置
      B. 稍后15分钟 → 弹窗关闭 → 15分钟后再次弹出
      C. 跳过 → 弹窗关闭 → 计时器正常重置（不记录）
      D. 30秒无操作 → 等同跳过
  → 宠物状态更新（已喝水 → 正常/达标状态）
```

### 6.3 离桌场景流程（手机接管）

```
用户关闭桌面应用 / 手动切换到手机模式
  → 手机端接管提醒
  → 手机收到推送通知
  → 点击通知 → 打开 App → 快速记录
  → 数据同步回 Firebase
  → 重新打开桌面端 → 自动拉取最新数据
```

## 七、非功能需求

### 7.1 性能要求

| 项目 | 要求 |
|---|---|
| 桌面端内存占用 | < 100MB（常驻后台状态） |
| 提醒弹窗响应延迟 | < 500ms |
| Firebase 同步延迟 | < 3秒（正常网络） |
| 应用启动时间 | < 3秒 |

### 7.2 兼容性

| 平台 | 要求 |
|---|---|
| macOS | 12（Monterey）及以上 |
| iOS | 16 及以上 |
| Windows | 暂不支持（V1 仅 macOS） |

### 7.3 权限要求

| 权限 | 用途 | 申请时机 |
|---|---|---|
| 通知权限 | 提醒弹窗（手机端） | 首次启动时申请 |
| 开机启动权限 | 桌面端自启 | 初始化完成时引导用户开启 |
| 网络权限 | Firebase 同步 | 无需显式申请 |

### 7.4 隐私与数据

- 所有用户数据存储在 Firebase，用户可随时删除账号及数据
- 不收集设备唯一标识符（IDFA 等）
- V1 不做数据分析上报，MVP 阶段数据仅用于功能运行
- 用户年龄段等敏感信息仅存储在用户自己的 Firebase 文档中，不作聚合分析
